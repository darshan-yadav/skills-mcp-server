# Project Manager Review Checklist

## 1. Scope commitment

- [ ] In-scope and out-of-scope both written.
- [ ] The outcome (what "done" looks like) is demonstrable, not activity-based.
- [ ] Non-goals explicit.

## 2. Milestone quality

- [ ] Each milestone is a demonstrable state.
- [ ] Each has a definition of done.
- [ ] Dependency graph visible; critical path marked.
- [ ] No milestone described only as "X complete" without what "complete" means.

## 3. Estimates & schedule

- [ ] Best / expected / worst ranges per milestone.
- [ ] Buffer exists and is named, not hidden.
- [ ] Worst-case date is tracked separately from expected.
- [ ] Re-estimation cadence defined.

## 4. RACI

- [ ] One Accountable per milestone.
- [ ] No milestone is owned by "the team" or a channel.
- [ ] Consulted list is scoped (not the whole org).
- [ ] Informed list covers actually-affected stakeholders.

## 5. Dependencies

- [ ] Cross-team dependencies named with the *receiving* team's tracking ID.
- [ ] Dated commitments from dependency owners (not "they'll do it").
- [ ] Re-confirmation cadence exists.

## 6. Risk register

- [ ] Produced from a premortem, not from memory alone.
- [ ] Each risk has likelihood × impact.
- [ ] Each risk has a **trigger** (what we'll watch).
- [ ] Each risk has **mitigation** (what we're doing now).
- [ ] Each risk has **contingency** (what we do if it fires).
- [ ] Each risk has an owner.
- [ ] Updated at least weekly.
- [ ] Closed-risk log retained with lessons.

## 7. Status reporting

- [ ] RAG used with cited evidence for each colour.
- [ ] "Green" is backed by completed milestones + no blockers.
- [ ] "Amber" names the blocker, the mitigation, and the deadline.
- [ ] "Red" is followed by a specific recovery plan.
- [ ] Status leads with decisions needed + risks, not progress narrative.
- [ ] Written status exists before any status meeting.

## 8. Change control

- [ ] Every scope / schedule / owner change logged.
- [ ] Each change records who decided and what compensated.
- [ ] Changes are visible to stakeholders, not buried in a doc.

## 9. Meeting discipline

- [ ] Recurring meetings have a death-condition.
- [ ] Agendas are decisions + blockers, not round-robin.
- [ ] Decision log exists.

## 10. Launch readiness

- [ ] Engineering, QA, SRE/Ops, Support, Legal, Comms, GTM, Product each have a checklist.
- [ ] Each has a named owner.
- [ ] Go/no-go decision authority named.
- [ ] Rollback authority named.
- [ ] Post-launch review scheduled before launch.

## 11. Close-out

- [ ] Retro / lessons planned.
- [ ] Estimate vs. actual captured.
- [ ] Risks fired: mitigation review.
- [ ] Risks didn't fire: was it management or luck?

## Comment style

- Lead with "this will cause a surprise at milestone M" or "this risk has no trigger — it's a worry."
- Severity on every comment: **blocker**, **non-blocker**, **nit**.
- Surface hidden assumptions as questions.
- Praise at least one thing.
